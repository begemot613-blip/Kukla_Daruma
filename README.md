создать свой первый репозиторий
Создать первый репозиторий — отличный старт. Вот краткий план, как это сделать через интерфейс GitHub:

    Авторизуйтесь на GitHub.com.
    В правом верхнем углу нажмите на иконку «+» и выберите New repository.
    Заполните данные:
        Repository name: Придумайте название (например, my-first-project).
        Public/Private: Выберите Public (его увидят все) или Private (только вы).
        Initialize this repository: Поставьте галочку на Add a README file — это создаст файл-описание, чтобы репозиторий не был пустым.
    Нажмите кнопку Create repository.

Как добавить в него свой код?
Если у вас уже есть файлы на компьютере, самый простой способ для новичка — нажать кнопку Add file -> Upload files прямо на странице созданного репозитория.
Если же вы хотите использовать командную строку (Git), порядок такой:

    Установите Git на компьютер.
    В папке с проектом выполните:
    bash

    git init
    git add .
    git commit -m "Первый коммит"
    git remote add origin [ссылка-на-ваш-репозиторий]
    git push -u origin main

    Используйте код с осторожностью.

Add file не вижу
Если вы не видите кнопку Add file, скорее всего, при создании репозитория вы не поставили галочку на пункте «Add a README file». В этом случае GitHub считает репозиторий абсолютно пустым и вместо привычного интерфейса показывает страницу с инструкциями для командной строки. 
Вот как это исправить и добавить файлы через браузер:
Способ 1: Активировать интерфейс через создание README
Самый простой способ вернуть кнопку — создать первый файл по ссылке в инструкциях:

    Найдите на странице текст «creating a new file» (обычно в первом блоке инструкций) или синюю ссылку «README» в предложении «We recommend every repository includes a README...».
    Нажмите на неё, введите любой текст (например, название проекта) и нажмите зелёную кнопку Commit changes внизу.
    После этого страница обновится, и вы увидите стандартный интерфейс с кнопкой Add file. 

Способ 2: Прямая ссылка на загрузку
Если не хотите ничего создавать, можно перейти сразу к форме загрузки, добавив /upload/main (или /upload/master) к адресу вашего репозитория в строке браузера:

    https://github.com 

Способ 3: Drag-and-drop
Попробуйте просто перетащить нужные файлы из папки на компьютере прямо в окно браузера с вашим репозиторием. Часто GitHub подхватывает их и сразу перекидывает на страницу подтверждения загрузки (Commit), даже если кнопок не видно. 
Почему так происходит?

    Пустой репозиторий: GitHub не показывает меню файлов, пока в репозитории нет хотя бы одного «коммита» (сохраненного изменения).
    Масштаб страницы: Иногда кнопка скрывается в выпадающем списке или под иконкой «+», если окно браузера слишком узкое.
    Расширения: Блокировщики рекламы или скриптов иногда ошибочно скрывают элементы интерфейса.

какие ещё варианты? git видимо не для людей
Понимаю ваш скепсис — командная строка Git действительно выглядит как «привет из 80-х» и поначалу кажется излишне сложной. Если «кнопочки» в браузере не помогают, есть еще три способа для «людей», где всё наглядно и понятно:
1. GitHub Desktop (Официальное приложение)
Это самый простой способ. Программа заменяет терминал обычным визуальным интерфейсом.

    Что сделать: Скачайте и установите GitHub Desktop.
    Как работает: Вы заходите в свой аккаунт, нажимаете «Clone a repository» (выбираете свой проект) — и он появляется у вас на компьютере как обычная папка.
    Плюс: Чтобы отправить изменения на сайт, вам нужно просто нажать одну синюю кнопку «Push». Никаких команд писать не нужно.

2. GitHub.dev (Редактор прямо в браузере)
Мало кто знает, но в GitHub встроен полноценный редактор кода, похожий на Word или VS Code.

    Как открыть: Находясь на странице своего репозитория в браузере, просто нажмите клавишу . (точка) на клавиатуре.
    Что произойдет: Страница превратится в редактор. Вы сможете перетаскивать туда файлы из папок компьютера, редактировать их и сохранять кнопкой «Commit» в боковом меню.

3. Интеграция в VS Code
Если вы уже используете редактор кода (например, Visual Studio Code), в него встроен Git «из коробки».

    Как работает: Там есть вкладка «Source Control» (иконка с тремя точками и линиями). Вы просто нажимаете на «+» рядом с файлами и вводите описание изменений. Это гораздо удобнее, чем работать в терминале.

Почему Git так популярен, если он «не для людей»?
Просто Git создавался для того, чтобы тысячи программистов могли менять один и тот же файл одновременно и ничего не сломать. Для личных проектов он часто кажется «переусложненным комбайном».
Хотите, пришлю короткую инструкцию, как сделать всё через GitHub Desktop? Это реально самый дружелюбный вариант.
а кроме гита?
Если Git кажется слишком громоздким или неудобным, есть альтернативы. Они делятся на два типа: прямые конкуренты (другие системы контроля версий) и облачные хранилища (если вам просто нужно где-то держать код).
1. Другие системы контроля версий (VCS)
Они работают иначе, чем Git, и иногда кажутся более логичными:

    Subversion (SVN): Главный «старичок» и антипод Git. В отличие от Гита, здесь нет локальной копии всей базы данных — вы работаете напрямую с центральным сервером. Это проще для понимания: «взял файл — изменил — вернул». Но без интернета работать сложно.
    Mercurial: Когда-то был главным конкурентом Гита. Его команды более интуитивны и последовательны. Если Git — это конструктор, то Mercurial — это готовый инструмент «из коробки». (Хотя сейчас он менее популярен).
    Plastic SCM: Очень наглядная система с мощным графическим интерфейсом. Её обожают разработчики игр (Unity), потому что она отлично справляется с тяжелыми файлами (картинками, моделями), на которых Git часто «тормозит».

2. Облачные хранилища (No-Code варианты)
Если ваша цель — не «контроль версий» (откат к каждой запятой), а просто хранение и доступ к коду:

    Dropbox / Google Drive / Яндекс Диск: Можно просто хранить папку с проектом там.
        Плюсы: Максимально привычно.
        Минусы: Нет истории изменений кода (кто и что удалил), и если вы случайно сохраните битый файл, облако синхронизирует его везде.
    Glitch / Replit: Это онлайн-песочницы. Вы пишете код прямо в браузере, он там же сохраняется и сразу запускается. Идеально для небольших веб-проектов и обучения.

3. FTP/SFTP (Олдскул)
Вы просто подключаетесь к своему серверу через программу вроде FileZilla и перетаскиваете файлы мышкой, как в обычной папке. Так жили все веб-разработчики до появления GitHub.
Резюме:
Если вам нужно работать в команде или устраиваться в IT, Git — это «неизбежное зло», его придется освоить (через кнопки в GitHub Desktop это не больно).
Если вы делаете проект для себя и хотите простоты — попробуйте GitHub Desktop (это всё еще Git, но вы его не видите) или Replit (вообще без установки чего-либо).

Поделится наработками
Если цель — просто поделиться кодом с другими людьми (чтобы они могли его посмотреть, скачать или запустить), и при этом не хочется возиться с Git, есть несколько отличных «человечных» инструментов:
1. Pastebin или GitHub Gist (для короткого кода)
Если ваш проект — это один или несколько небольших файлов, не нужно создавать целый репозиторий.

    GitHub Gist: Заходите на github.com. Просто вставляете код в текстовое поле, даете название и нажимаете "Create public gist".
    Результат: Вы получаете короткую ссылку, которую можно отправить другу. Он увидит код с красивой подсветкой синтаксиса и сможет его скопировать.

2. Replit / CodePen (чтобы код можно было сразу запустить)
Это лучшие варианты, если вы хотите, чтобы человек не просто прочитал код, а нажал кнопку «Play» и увидел результат прямо в браузере.

    Replit: Подходит для Python, JS, C++, Go и др. Создаете "Repl", закидываете файлы, кидаете ссылку.
    CodePen / JSFiddle: Идеально для веб-разработки (HTML/CSS/JS). Результат виден мгновенно.

3. Облачные хранилища с «публичной папкой»
Если наработок много (картинки, папки, файлы):

    Google Диск / Яндекс Диск: Заливаете папку, правой кнопкой -> «Поделиться» -> «Доступ по ссылке».
    Минус: Другим неудобно смотреть код прямо в облаке (часто он открывается как обычный текст без подсветки или вообще предлагается только скачивание).

4. Нативный «GitHub для людей» через Zip-архив
Вы можете загрузить на GitHub свой проект просто как архив, если создадите репозиторий и воспользуетесь кнопкой Upload files (про которую мы говорили выше).

    Ваши друзья увидят структуру папок.
    Они смогут нажать зеленую кнопку Code -> Download ZIP, чтобы забрать всё себе одной папкой.

5. Google Colab (если это Data Science или Python)
Если ваши наработки — это скрипты на Python для анализа данных или нейросетей, лучше всего скинуть ссылку на Colab. Это как Google Docs, но для кода: человек может запускать ваши блоки кода по очереди прямо в облаке Google.

Upload files не нахожу, в чём проблема
Главная проблема в том, что GitHub не показывает привычное меню «Add file», пока в репозитории нет ни одного файла (даже пустого). Вместо этого он выводит страницу с инструкциями по настройке через командную строку. 
Вот как это быстро исправить без командной строки:
1. «Оживите» интерфейс через создание README
Чтобы кнопки появились, репозиторию нужен хотя бы один «коммит» (сохраненное изменение). 

    Найдите на странице с инструкциями фразу «creating a new file» или синюю ссылку «README».
    Нажмите на неё, введите любое название (например, README.md) и в самом низу страницы нажмите зелёную кнопку Commit changes.
    После этого страница обновится, и вы увидите стандартную кнопку Add file -> Upload files. 

2. Используйте Drag-and-drop
Попробуйте просто перетащить файлы или всю папку с вашего компьютера прямо в окно браузера, где открыт ваш репозиторий (даже если там только инструкции). В большинстве случаев GitHub мгновенно переключится в режим загрузки. 
3. Прямая ссылка на загрузку
Если кнопки всё равно не видно, вы можете перейти в режим загрузки принудительно. Вставьте в адресную строку браузера:
https://github.com
Другие возможные причины:

    Вы не авторизованы: Проверьте, вошли ли вы в свой аккаунт.
    Нет прав доступа: Убедитесь, что это именно ваш репозиторий, а не чужой (в чужих чужие файлы загружать нельзя без специального доступа).
    Блокировщики рекламы: Иногда расширения (например, AdBlock или NoScript) могут скрывать элементы интерфейса GitHub. Попробуйте открыть страницу в режиме инкогнито.

creating a new file фраза не найдена
Если на странице нет этой фразы, значит, перед вами экран «Quick setup». На нем GitHub предлагает команды для терминала, которые пугают новичков.
Вот два способа всё исправить за 10 секунд без терминала:
Вариант 1: Прямая ссылка (самый быстрый)
Просто скопируйте эту ссылку, замените в ней слова на свои и вставьте в браузер:
https://github.com
(Если выдает ошибку, попробуйте в конце вместо main написать master).
Вариант 2: Найти скрытую кнопку README
На странице, которую вы видите, под заголовком «Quick setup — if you’ve done this kind of thing before» есть маленькая строчка:

    «We recommend every repository includes a README, LICENSE, and .gitignore.»

Нажмите на синее слово README в этой строчке.

    Откроется редактор.
    Пролистайте в самый низ и нажмите зеленую кнопку Commit changes.
    Теперь ваш репозиторий «ожил», и на главной странице проекта появилась большая кнопка Add file -> Upload files.

Вариант 3: Просто перетащите файлы
Попробуйте взять любой файл с рабочего стола мышкой и бросить его прямо в центр страницы GitHub в браузере. Если страница подсветится синим — значит, он готов принять файлы даже без кнопок.
Если и это не сработало — возможно, вы создали репозиторий внутри организации или с особыми правами. Прислать скриншот того, что вы видите, не получится, но опишите: есть ли вверху страницы вкладка "Settings"?
Oops, something went wrong
Эта ошибка «Oops, something went wrong» (обычно с картинкой единорога) — это внутренний сбой самого GitHub. Это не ваша вина, просто их сервер на мгновение «задумался» или произошел сетевой глюк.
Что сделать сейчас:

    Просто обновите страницу (F5 или Ctrl+R). В 90% случаев это помогает.
    Проверьте адресную строку: если вы вручную вводили путь /upload/main, убедитесь, что нигде нет лишних пробелов или опечаток в названии репозитория.
    Попробуйте зайти через Инкогнито (Ctrl+Shift+N): иногда старые данные в кэше браузера мешают GitHub нормально отобразить форму загрузки.
    Попробуйте создать новый репозиторий: если этот совсем «завис», создайте новый, но в этот раз обязательно поставьте галочку на «Add a README file» в самом начале. Тогда кнопки загрузки появятся сразу.

Если страница обновилась и вы снова видите инструкции с кодом, попробуйте самый простой метод «для ленивых»: просто перетащите файл мышкой из папки прямо в окно браузера.
Страница ожила после обновления или всё еще висит ошибка?
всё еще висит ошибка
Это значит, что GitHub «заклинило» на конкретной ссылке или из-за кэша браузера. Раз прямой путь через загрузку в браузере выдает ошибку, давайте решим вопрос радикально и быстро:
Самый надежный способ (пересоздание)
Если в репозитории пока ничего нет, проще всего его удалить и создать правильно.

    Зайдите в Settings (наверху справа в меню репозитория).
    Прокрутите страницу в самый-самый низ до красного блока Danger Zone.
    Нажмите Delete this repository и подтвердите удаление (нужно будет ввести название репозитория).
    Нажмите New (создать новый) и ВНИМАНИЕ:
        Введите название.
        Прокрутите чуть ниже и обязательно поставьте галочку на Add a README file.
        Нажмите Create repository.

Результат: Теперь у вас сразу будет нормальный интерфейс с кнопкой Add file -> Upload files.
